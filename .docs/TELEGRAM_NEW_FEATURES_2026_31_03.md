# Изменения для Telegram-бота на 31.03.2026

Этот документ фиксирует, что именно нужно поменять в Telegram-боте после изменений в backend `Reflebot`.

Здесь описаны только практические последствия для bot-side логики. Часть backend-изменений не требует доработок в самом боте, потому что бот по-прежнему получает готовые `message`, `buttons`, `files` и `dialog_messages`.

## Что точно нужно поменять в Telegram-боте

### 1. Приглашение студента на курс по ссылке

Теперь backend поддерживает регистрацию студента по invite-ссылке курса.

Что изменилось:
- после создания курса backend показывает пригласительную ссылку для студентов;
- ссылка ведёт в Telegram-бот через deep link;
- при входе по такой ссылке backend может:
  - привязать уже существующего студента к новому курсу;
  - обновить `telegram_id`, если нужно;
  - попросить ввести ФИО, если студент ещё не существует.

Что должен сделать Telegram-бот:
- уметь читать payload из Telegram deep link `/start <payload>`;
- передавать этот payload в backend при login как `invite_token`;
- не терять обычный сценарий `/start` без payload.

Практически это означает:
- раньше login вызывался только с `telegram_id`;
- теперь, если у пользователя есть payload после `/start`, его нужно добавить в body login-запроса.

Ожидаемое поведение:
- если backend вернул обычный login response, бот просто рендерит его как раньше;
- если backend вернул сообщение `Введите своё ФИО...` и `awaiting_input=true`, бот просто ждёт текст;
- если backend вернул `Вы успешно зарегистрировались на курс ...`, бот просто показывает это сообщение без кнопок.

Что на стороне бота не нужно делать:
- не нужно самостоятельно разбирать курс из ссылки;
- не нужно самим записывать студента на курс;
- не нужно хранить invite state локально.

Всё это делает backend.

### 2. В аналитике `dialog_messages` теперь могут содержать кнопки

Раньше финальные действия обычно приходили только в основном `response.buttons`.

Теперь в сценарии аналитики, после вывода всех файлов и сообщений студента, backend добавляет ещё один элемент в `dialog_messages`:
- `message = "Что делаем далее?"`
- `buttons = [...]`

Это сделано для экранов просмотра рефлексии и ответов студента, чтобы после истории сразу снова показать рабочее меню.

Что должен сделать Telegram-бот:
- поддержать рендер `buttons` не только у основного ответа, но и у каждого элемента `dialog_messages`;
- если у элемента `dialog_messages[i]` есть `message`, бот отправляет это сообщение;
- если у него есть `buttons`, бот прикрепляет их именно к этому сообщению;
- если у него есть `files`, бот отправляет их после этого сообщения.

Если бот этого не сделает:
- история рефлексии покажется;
- но финальные кнопки `Аналитика`, `Ближайшая лекция` и admin-кнопки после истории не появятся.

Это единственное изменение в контракте рендера, которое действительно требует доработки bot-side логики.

## Что менять в Telegram-боте, скорее всего, не нужно

### 3. Дедлайн лекции

Теперь у лекции есть `deadline`, а backend:
- добавляет дедлайн в текст сообщения студенту;
- не даёт начать запись кружка после дедлайна.

Для Telegram-бота это не требует новой бизнес-логики.

Почему:
- бот просто показывает текст, который пришёл от backend;
- проверку дедлайна делает backend при нажатии на кнопку `Записать кружок`;
- если дедлайн истёк, backend сам вернёт готовое сообщение об ошибке.

Что важно:
- бот не должен сам сравнивать время и решать, можно ли записывать кружок;
- бот не должен сам скрывать кнопку по времени;
- бот просто отправляет action в backend и отображает ответ.

### 4. Вопросы по умолчанию

Теперь после создания курса backend может добавить стандартные вопросы к лекциям по отдельной кнопке.

Для Telegram-бота здесь ничего нового нет:
- backend сам отдаёт кнопку;
- бот просто показывает её;
- при нажатии отправляет action обратно;
- backend сам создаёт вопросы и возвращает новый экран.

От бота не требуется:
- генерировать вопросы;
- выбирать случайный вопрос;
- хранить состояние этого сценария.

### 5. Новая кнопка `Курсы`

Для администратора в главном меню появилась новая кнопка:
- `Курсы`

Также появился новый сценарий:
- список курсов с пагинацией;
- карточка курса с названием и пригласительной ссылкой;
- кнопка `Назад`.

Для Telegram-бота это обычно не требует отдельной доработки, если он уже:
- рендерит `buttons`;
- передаёт `callback_data` обратно в backend без изменений;
- умеет работать с пагинацией и `Назад`.

То есть это backend-driven изменение:
- новые `action` приходят автоматически;
- бот просто должен не ломать общий механизм callback-кнопок.

## Итого

### Обязательные изменения в Telegram-боте

1. Передавать `invite_token` из deep link `/start <payload>` в login endpoint backend.
2. Поддержать `buttons` внутри `dialog_messages`.

### Изменения, которые обычно не требуют bot-side логики

1. Дедлайн лекции.
2. Стандартные вопросы.
3. Новая admin-кнопка `Курсы`, если бот уже корректно рендерит обычные backend-driven кнопки.

## Рекомендуемый короткий чеклист для Telegram-бота

- Проверить обработку `/start` с payload.
- Проверить, что login-запрос умеет отправлять `invite_token`.
- Проверить, что `dialog_messages[].buttons` рендерятся в Telegram.
- Проверить, что экран истории рефлексии теперь заканчивается сообщением `Что делаем далее?` с кнопками.
- Проверить, что новые admin-кнопки отображаются без хардкода.

## Что не нужно делать в Telegram-боте

- не реализовывать invite-логику самостоятельно;
- не валидировать дедлайн локально;
- не генерировать стандартные вопросы;
- не разбирать новые admin action вручную, если общий callback-flow уже работает;
- не хранить дополнительное состояние там, где backend уже управляет сценарием сам.
