# FEATURES

## Broker delivery idempotency

Нужно добавить надёжную идемпотентность для обработки broker-команды `send_reflection_prompt` по `delivery_id`.

Почему это нужно:

- если prompt уже был успешно отправлен в Telegram, а `success-result` не удалось опубликовать в RabbitMQ, бот не должен повторно отправлять тот же prompt;
- без сохранённого факта "этот `delivery_id` уже был успешно доставлен" бот не сможет отличить повторную обработку после рестарта или повторного получения сообщения;
- текущая логика может лишь снизить риск дублей в рамках одного запуска процесса, но не гарантирует защиту от дублей надёжно.

Что нужно сделать позже:

- добавить persistent storage для статуса доставки по `delivery_id`;
- сохранять как минимум факт успешной отправки в Telegram и связанный `telegram_message_id`;
- при повторной обработке сначала проверять уже известный `delivery_id`, а затем решать, отправлять ли prompt повторно или только допубликовать result event.
