# Правила генерации спеки

## Назначение

Этот документ фиксирует, как в проекте создавать новые спеки в `.kiro/specs/<feature-name>/`.

Текущий принятый формат спеки состоит из 3 файлов:

- `requirements.md`
- `design.md`
- `tasks.md`

Писать их нужно именно в таком порядке:

1. сначала `requirements.md`
2. потом `design.md`
3. потом `tasks.md`

Причина простая:

- `requirements.md` отвечает на вопрос "что должно уметь решение"
- `design.md` отвечает на вопрос "как именно это будет устроено"
- `tasks.md` отвечает на вопрос "в каком порядке это реализовать"

---

## Где создавать спеку

Новая спека создаётся в отдельной папке:

```text
.kiro/specs/<feature-name>/
```

Правила имени папки:

- использовать `kebab-case`
- имя должно коротко описывать фичу
- не использовать слишком общие названия вроде `new-feature` или `refactor`

Хорошие примеры:

- `telegram-bot-full-workflow`
- `reflection-prompt-delivery-workflow`

---

## Общие правила для всех 3 файлов

- писать на русском языке
- использовать предметные термины проекта: `System`, `Bot`, `Admin`, `Teacher`, `Student`, `Course`, `Lection`, `Reflection`
- быть конкретным и проверяемым
- не писать абстрактные формулировки без критерия проверки
- не смешивать требования, дизайн и план реализации в одном разделе
- сохранять трассировку: задачи должны ссылаться на требования, дизайн должен покрывать требования

Что важно:

- `requirements.md` не должен превращаться в список классов и файлов
- `design.md` не должен превращаться в backlog
- `tasks.md` не должен быть пересказом требований

---

## Правила для `requirements.md`

### Цель документа

`requirements.md` фиксирует ожидаемое поведение системы и границы фичи.

### Рекомендуемая структура

```md
# Документ требований: <название фичи>

## Введение

## Глоссарий

## Требования
```

### Что обязательно должно быть

#### 1. Введение

Во введении нужно кратко описать:

- зачем нужна фича
- какую проблему она решает
- какие архитектурные ограничения важны сразу

#### 2. Глоссарий

В глоссарии фиксируются термины, которые дальше будут использоваться в тексте.

Нужно добавлять только реально используемые сущности.

#### 3. Требования

Каждое требование оформляется так:

```md
### Требование N: Название

**User Story:** Как <роль>, я хочу <цель>, чтобы <ценность>.

#### Критерии приемки

1. THE System SHALL ...
2. WHEN ...
3. IF ... THEN THE System SHALL ...
```

### Правила написания требований

- одно требование = одна логическая capability
- у требования должно быть короткое, предметное название
- у каждого требования должна быть `User Story`
- у каждого требования должны быть критерии приёмки
- критерии должны быть нумерованы
- в критериях использовать строгие формулировки:
  - `THE System SHALL`
  - `WHEN ... THE System SHALL`
  - `IF ... THEN THE System SHALL`
  - `AFTER ... THE System SHALL`

### Чего нельзя делать в `requirements.md`

- нельзя писать "создать файл", "добавить сервис", "внедрить DI"
- нельзя описывать конкретные имена методов как часть требования без необходимости
- нельзя писать vague-формулировки вроде:
  - "система должна работать быстро"
  - "система должна быть удобной"
  - "должно быть оптимально"

Вместо этого надо писать проверяемо:

- "THE System SHALL выполнять bounded scan батчами"
- "THE System SHALL не сканировать всю историческую таблицу на каждом запуске"

### Что обязательно проверить при написании

- можно ли по тексту требования написать тест
- понятно ли, что считается success/failure
- есть ли отдельные требования для:
  - права доступа
  - идемпотентность
  - обработка ошибок
  - производительность, если она важна
  - настройки через `Settings`, если фича инфраструктурная

## Правила для `design.md`

### Цель документа

`design.md` описывает техническое решение, которое удовлетворяет требованиям.

### Рекомендуемая структура

```md
# Технический дизайн: <название фичи>

## Обзор

## Архитектура

## Ключевые решения

## Модели данных

## Структура модулей

## Потоки выполнения / state machine / error handling
```

### Что обязательно должно быть

#### 1. Обзор

Краткое описание выбранного решения и его принципов.

#### 2. Архитектура

Нужно описать:

- основные компоненты
- как они взаимодействуют
- какие внешние системы участвуют

Если фича нетривиальная, лучше добавить ASCII-схему.

#### 3. Ключевые решения

Здесь фиксируются важные выборы:

- почему выбран именно этот подход
- почему не выбран альтернативный подход
- какие trade-off'ы приняты

#### 4. Модели данных

Если появляются новые таблицы, enum'ы, payload'ы или контракты, они должны быть описаны здесь.

#### 5. Структура модулей

Нужно перечислить:

- в каких модулях будет реализация
- какие новые файлы добавляются
- какая ответственность у каждого компонента

### Правила написания `design.md`

- дизайн должен покрывать требования, но не копировать их дословно
- дизайн должен быть достаточно конкретным, чтобы по нему можно было разбить задачу на реализацию
- если есть фоновая обработка, брокер, cron, Celery, consumer, webhook, state machine — это должно быть явно описано
- если есть производительные места, нужно описать:
  - какие запросы будут bounded
  - где будут индексы
  - где будет batch processing

### Что нельзя делать в `design.md`

- нельзя писать только общие слова без структуры
- нельзя превращать его в TODO list
- нельзя пропускать error handling, если фича асинхронная или распределённая

## Правила для `tasks.md`

### Цель документа

`tasks.md` переводит дизайн в поэтапный план реализации.

### Рекомендуемая структура

```md
# Implementation Plan: <название фичи>

## Overview

## Tasks
```

### Формат задач

Используется markdown checklist:

```md
- [ ] 1. Название крупного этапа
  - [ ] 1.1 Подзадача
  - [ ] 1.2 Подзадача
    - _Requirements: 1.1, 1.2, 3.4_
```

### Правила декомпозиции

- верхний уровень — крупные этапы
- внутри — подзадачи, которые можно реализовать и проверить
- каждая подзадача должна быть достаточно маленькой, чтобы быть понятной и измеримой
- задачи должны идти в порядке реализации
- сначала foundation:
  - settings
  - enums
  - schemas
  - models
  - migrations
- потом infrastructure и repositories
- потом services и use cases
- потом handlers / routers / consumers / tasks
- потом tests
- потом integration / smoke / docs

### Что обязательно должно быть в `tasks.md`

- краткий `Overview`
- ключевые принципы реализации
- указание стека, если он важен
- ссылки `_Requirements: ..._` у подзадач
- checkpoint tasks там, где нужен промежуточный прогон тестов

### Как писать хорошие задачи

Плохая задача:

- "Сделать уведомления"

Хорошая задача:

- "Создать `NotificationDeliveryRepository`"
- "Реализовать `get_pending_batch`"
- "Реализовать `mark_sent` и `mark_failed`"
- "Написать unit tests для переходов статусов"

### Что нельзя делать в `tasks.md`

- нельзя делать задачи слишком крупными
- нельзя оставлять задачи без связи с требованиями
- нельзя смешивать реализацию и бизнес-требования в одном bullet

---

## Рекомендуемый порядок генерации спеки

### Шаг 1. Сформулировать границы фичи

Нужно ответить:

- что именно добавляем
- чего точно не делаем в этой спеки
- какие внешние зависимости участвуют

### Шаг 2. Написать `requirements.md`

Сначала формулируется поведение системы.

На этом шаге нельзя сразу прыгать в файлы и классы.

### Шаг 3. Написать `design.md`

После утверждения требований описывается:

- архитектура
- данные
- компоненты
- потоки
- отказоустойчивость

### Шаг 4. Написать `tasks.md`

После дизайна делается реализационный план:

- по этапам
- с dependency order
- с requirements traceability

---

## Минимальный чеклист качества спеки

Перед тем как считать спеку готовой, нужно проверить:

- у фичи есть отдельная папка в `.kiro/specs`
- есть все 3 файла: `requirements.md`, `design.md`, `tasks.md`
- в `requirements.md` есть введение, глоссарий и критерии приёмки
- в `design.md` есть архитектура, ключевые решения и структура модулей
- в `tasks.md` есть нумерованные checklist-задачи
- задачи ссылаются на требования
- требования проверяемы
- дизайн покрывает требования
- задачи покрывают дизайн

---

## Практическое правило проекта

В этом проекте спеку нужно писать так, чтобы по ней можно было:

1. понять поведение системы без чтения кода
2. спроектировать решение без двусмысленности
3. реализовать фичу по шагам
4. проверить готовность через тесты, integration и smoke

Если документ не помогает сделать одно из этого, значит спека слишком размыта или неполна.
