# Изменения для Telegram-бота: код курса и `join_course`

Этот документ описывает новые изменения в backend, которые влияют на Telegram-бот.

Главная идея:
- логика пригласительной ссылки удалена;
- вместо неё теперь используется код курса;
- бот больше не должен запускать регистрацию студента через deep link payload;
- для записи на дополнительные курсы появилась команда `join_course`.

## Что поменялось в backend

### 1. Регистрация студента теперь идёт по коду курса

Старый сценарий с deep link и `invite_token` больше не является основным.

Теперь новый пользователь, которого backend не нашёл по `telegram_username` в ролях:
- не получает `404`;
- получает сообщение:
  - `Привет студент, введи код курса.`
- backend ставит `awaiting_input = true`
- дальше ждёт текст с кодом курса через обычный `POST /actions/text`

После ввода кода backend:
- определяет курс по коду;
- возвращает новый шаг:
  - `Вы выбрали курс {course_name}.`
  - `Введите своё ФИО.`
- снова ждёт текст через `POST /actions/text`

После ввода ФИО backend:
- создаёт студента;
- записывает студента на курс и на все лекции курса;
- возвращает:
  - `Вы успешно записались на курс: {course_name}.`

### 2. Появилась команда `join_course`

Команда нужна для уже существующего студента, чтобы записаться на ещё один курс.

Сценарий такой:
- студент отправляет команду `join_course`
- backend отвечает:
  - `Введите код курса.`
- backend ждёт следующий текст
- студент отправляет код курса
- backend записывает его на выбранный курс и на все лекции курса
- backend отвечает:
  - `Вы успешно записались на курс: {course_name}.`

### 3. В карточках курса теперь показывается код, а не ссылка

После создания курса и в admin-экране курса backend теперь показывает:
- код курса

Вместо:
- пригласительной ссылки на бота

Для Telegram-бота это означает, что никакой дополнительной логики тут не нужно:
- бот просто рендерит текст, который пришёл от backend.

## Что нужно изменить в Telegram-боте

### 1. Убрать зависимость от deep link invite-flow

Что больше не нужно:
- специальная логика чтения invite payload из `/start <payload>`
- передача `invite_token` в login endpoint как обязательной части нового student flow
- ожидание, что запись на курс начнётся только через ссылку

Если у бота уже есть код для deep link:
- его можно оставить как legacy;
- но новый сценарий регистрации на курс больше строится не на нём.

### 2. Обычный `/start` должен всегда идти в login endpoint

Бот должен как и раньше вызывать:
- `POST /api/reflections/auth/{telegram_username}/login`

Body минимально:
```json
{
  "telegram_id": 123456789
}
```

Важно:
- теперь backend может вернуть не только меню ролей;
- он может вернуть экран ввода кода курса для нового пользователя.

То есть бот после `/start` должен:
- не предполагать, что login всегда заканчивается кнопками;
- уметь корректно обработать `awaiting_input = true`.

### 3. Не перехватывать `join_course` локально как отдельную бизнес-логику

Новый сценарий уже реализован на backend.

Что должен делать бот:
- если пользователь отправил `join_course` или `/join_course`, передать этот текст в:
  - `POST /api/reflections/actions/text`
- backend сам определит, что это команда присоединения к курсу;
- backend сам вернёт следующий экран.

Что бот не должен делать:
- не должен сам просить код курса;
- не должен сам валидировать код;
- не должен сам записывать студента на курс.

### 4. Поддержать простой text-driven flow после login

Теперь после `/start` возможен такой сценарий:
1. бот отправил login;
2. backend ответил `Привет студент, введи код курса.`;
3. backend выставил `awaiting_input = true`;
4. бот ждёт обычный текст от пользователя;
5. бот отправляет этот текст в `/actions/text`;
6. backend отвечает `Вы выбрали курс ... Введите своё ФИО.`;
7. бот снова ждёт текст;
8. бот отправляет ФИО в `/actions/text`;
9. backend либо записывает пользователя на курс, либо возвращает ошибку валидации.

То есть bot-side поведение тут должно быть стандартным backend-driven:
- показать текст с запросом кода;
- дождаться кода;
- переслать его в backend;
- показать текст с запросом ФИО;
- дождаться ФИО;
- переслать его в backend.

### 5. Команда `join_course` должна работать для уже зарегистрированного студента

Ожидаемое поведение Telegram-бота:
- пользователь пишет `join_course` или `/join_course`;
- бот отправляет это как обычный текст в backend;
- backend возвращает `Введите код курса.`;
- бот показывает это сообщение;
- следующий текст пользователя тоже уходит в `/actions/text`.

## Что менять в Telegram-боте, скорее всего, не нужно

### 1. Рендер курса после создания

После создания курса backend сам показывает код курса в тексте.

Боту не нужно:
- генерировать код;
- копировать код в отдельную кнопку;
- распознавать код в тексте.

### 2. Проверка корректности кода курса

Боту не нужно:
- валидировать формат кода;
- проверять подпись;
- понимать, к какому курсу он относится.

Если код неверный:
- backend сам вернёт ошибку.

## Рекомендуемый bot-side чеклист

1. Проверить, что `/start` всегда делает login даже без специальных payload.
2. Проверить, что bot корректно рендерит login response с `awaiting_input = true`.
3. Проверить, что новый пользователь после `/start` может:
   - ввести код курса;
   - получить запрос на ФИО;
   - ввести ФИО;
   - получить успешную запись на курс.
4. Проверить, что существующий студент может отправить `join_course`.
5. Проверить, что после `join_course` backend просит код курса и успешно записывает на новый курс.
6. Проверить, что бот больше не зависит от deep link invite-сценария для основной регистрации на курс.

## Коротко

Теперь Telegram-боту нужно поддерживать два основных user flow:

1. `/start`
- обычный login
- если пользователь новый, backend просит код курса, а потом ФИО

2. `join_course`
- backend просит код курса
- после ввода кода backend записывает студента на курс

Во всех случаях:
- backend остаётся источником правды;
- бот только отправляет текст в backend и рендерит полученный ответ.
